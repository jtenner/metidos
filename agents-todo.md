# Agents Todo

This file is the execution backlog for replacing `@openai/codex-sdk` with Pi in `jt-ide`.

## Rules

- Remove completed todo items from this document altogether. Do not leave completed items in place as checked, archived, or struck through entries.
- Keep the `Risks` and `Blockers` sections current before adding, reordering, or splitting slices.
- Add new slices only when they clearly map back to the migration requirements in [the research document](./docs/2026-04-09-pi-coding-agent-migration-research.md).

## Risks

- Hosted-provider compatibility risk. The Pi runtime adapter now runs real Jolt thread lifecycles through Bun successfully, but the committed coverage is still probe-backed plus focused smoke tests. Real hosted-provider auth, transport quirks, and long-running tool sessions still need follow-up coverage. See [1. Runtime Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#1-runtime-integration), [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), and [Recommended Phased Plan](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).
- Safety semantics risk. `unsafeMode` now has an explicit Pi-era meaning in code: safe threads keep worktree-scoped file/search/edit/write tools and cannot escalate Jolt-created child threads or cron jobs to unsafe mode, while unsafe threads also gain `bash` and unsafe child creation. That closes the biggest ambiguity, but it is still not equivalent to Codex sandbox/network semantics or a real container boundary. See [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), [SA09 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#sa09-implementation-status-in-jt-ide), [No built-in Codex-style sandbox policy](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-codex-style-sandbox-policy), and [6. Access Controls and Unsafe Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#6-access-controls-and-unsafe-mode).
- Feature parity risk. GitHub tools, sub-agents, plan mode, and web search are not native Pi features, so parity still requires explicit custom implementation and careful scope control even though Jolt’s own thread/cron/context/vm2 tool pack is now ported for the Pi runtime path. See [Pi Non-Goals That Matter for Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#pi-non-goals-that-matter-for-jolt), [JT08 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#jt08-implementation-status-in-jt-ide), [7. GitHub Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#7-github-integration), and [8. Agents and Plan Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#8-agents-and-plan-mode).
- Transcript parity risk. TL07 restores minimal `file_change` parity for successful Pi `edit` and `write` calls, but `web_search` and any richer Pi-extension-specific transcript kinds are still unresolved. See [TL07 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#tl07-implementation-status-in-jt-ide), [Current Codex-Specific Surface In Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#current-codex-specific-surface-in-jolt), and [5. Transcript/Event Mapping](./docs/2026-04-09-pi-coding-agent-migration-research.md#5-transcriptevent-mapping).
- Pi session lifecycle risk. DB03 now persists `piSessionId`, `piSessionFile`, and `piLeafEntryId`, but Jolt still only stores references to Pi’s JSONL session system and does not yet expose or stress-test branching, recovery, or UI-level session navigation. See [DB03 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#db03-implementation-status-in-jt-ide), [3. Pi already has a full session system](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-pi-already-has-a-full-session-system), and [2. Persistence and Thread Identity](./docs/2026-04-09-pi-coding-agent-migration-research.md#2-persistence-and-thread-identity).
- Provider auth and availability risk. UI05 now exposes provider-qualified model labels and model-specific thinking support in the browser, but the app still has no provider login, auth-health, or availability hints for users choosing across Pi-backed endpoints. See [UI05 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#ui05-implementation-status-in-jt-ide), [3. Provider/Model UI Migration](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-providermodel-ui-migration), and [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix).

## Blockers

- Choose the GitHub integration strategy: `gh` wrapper, REST/GraphQL extension, or hybrid. This blocks parity for the existing `GitHub` thread access toggle. See [No built-in GitHub connector](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-github-connector) and [7. GitHub Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#7-github-integration).
- Decide acceptable parity for agents and plan mode before implementation begins. Otherwise the migration will drift between “exact Codex parity” and “Pi-native simplification” mid-stream. See [8. Agents and Plan Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#8-agents-and-plan-mode) and [Open Questions](./docs/2026-04-09-pi-coding-agent-migration-research.md#open-questions-that-must-be-answered-before-full-migration).

## Todo Items

- [GH][10] - Implement a Pi-Native GitHub Tool Pack
Choose and build the post-Codex GitHub integration so the existing `GitHub` access toggle has a real Pi-era meaning, whether through `gh`, REST/GraphQL, or a hybrid. This slice is driven by [No built-in GitHub connector](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-github-connector), [7. GitHub Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#7-github-integration), and [What Jolt Still Must Build By Hand Even If It Uses Pi](./docs/2026-04-09-pi-coding-agent-migration-research.md#what-jolt-still-must-build-by-hand-even-if-it-uses-pi).

- [AG][11] - Decide and Implement Pi-Era Agents and Plan Mode
Either adopt Pi’s example subagent and plan-mode extensions or intentionally reduce scope, but make the decision explicit and map `agentsAccess` to the chosen behavior. This slice should follow [No built-in sub-agents or plan mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-sub-agents-or-plan-mode), [8. Agents and Plan Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#8-agents-and-plan-mode), and [Phase 6](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [CR][12] - Rewire Cron Execution to Pi Sessions
Keep the existing Jolt cron scheduler but change its execution path so cron-created threads resolve Pi models, Pi sessions, Pi tools, and Pi safety policy instead of Codex threads. This slice should implement [9. Cron Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#9-cron-integration), [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix), and [Phase 4](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [TM][13] - Replace Codex Telemetry with Pi Usage and Compaction Data
Delete the Codex JSONL scraping path and expose Pi-native usage, context, queue, and compaction data through Jolt’s thread detail and status surfaces. This slice should follow [6. Pi already has compaction and context usage concepts](./docs/2026-04-09-pi-coding-agent-migration-research.md#6-pi-already-has-compaction-and-context-usage-concepts), [10. Telemetry, Usage, and Compaction UI](./docs/2026-04-09-pi-coding-agent-migration-research.md#10-telemetry-usage-and-compaction-ui), and [Phase 3](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [UX][14] - Bind Pi Extension UI Interactions into the Browser App
Implement React-backed handlers for Pi extension prompts, confirmations, selections, editor requests, status updates, and widgets so extension-heavy workflows can operate inside the existing Jolt frontend. This slice is defined by [4. Pi already has built-in coding tools and a strong extension surface](./docs/2026-04-09-pi-coding-agent-migration-research.md#4-pi-already-has-built-in-coding-tools-and-a-strong-extension-surface), [12. Browser UI Bindings for Pi Extensions](./docs/2026-04-09-pi-coding-agent-migration-research.md#12-browser-ui-bindings-for-pi-extensions), and [Phase 6](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [RM][15] - Remove Codex SDK and Legacy Codex Paths
After Pi reaches acceptable parity, delete the Codex SDK dependency, Codex-specific runtime code, telemetry adapters, schema remnants, labels, and MCP bridge paths that no longer serve the product. This slice should be the final cleanup described in [Current Codex-Specific Surface In Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#current-codex-specific-surface-in-jolt), [Phase 7](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan), and [Recommendation](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommendation).
