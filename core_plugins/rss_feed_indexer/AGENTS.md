# AGENTS for RSS Feed Indexer

## Purpose

This first-party core plugin indexes RSS and OPML subscriptions into plugin-owned LanceDB vector storage and exposes the `RSS: Query` access group with the `rss_query` tool. It registers a daily app-local cron that reads the local operator's `list_url` setting, fetches approved feed URLs, embeds changed/new RSS items with Metidos embeddings, and upserts them into the plugin's current feed index.

It touches external network URLs only through `metidos.fetch`. The checked-in manifest intentionally declares the existing `unsafe` permission and `network.allow: ["https://**/**"]` so the local operator can configure arbitrary public HTTPS RSS or OPML hosts. Keep this only when the local operator accepts the all-domain review risk; fetched XML and feed content must be treated as untrusted.

## Source layout

- `metidos-plugin.json`: Plugin System v1 manifest reviewed by the local operator. It declares `unsafe` to permit all-domain HTTPS feed fetching with `https://**/**`; replace with literal origins in the app-data plugin copy if broad feed access is not acceptable.
- `index.ts`: plugin entry point; registers `metidos.cron({ key: "refresh_feeds" })` and `rss_query`.
- `README.md`: setup and usage notes.
- `AGENTS.md`: this guide.
- `.data/`: generated plugin data owned by Metidos; do not commit.
- `.logs/`: generated plugin logs when enabled; do not commit.
- `.data-bak-*`: generated reset backups; do not commit.

Root `node_modules/` is forbidden. Plugin code may import only `@metidos/plugin-api` and local relative files inside this plugin folder.

## Validation

1. Validate the manifest against `docs/metidos-plugin.schema.json` from the Metidos repository root.
2. Run `bun test src/bun/plugin/core-rss-feed-indexer-plugin.test.ts`.
3. Confirm every registered tool and cron is declared in the manifest:
   - tool: `rss_query`
   - cron: `refresh_feeds`
4. Confirm no root `node_modules/` exists.
5. Confirm all-domain `network.allow` is intentional and paired with `unsafe`, or replace it with literal HTTPS RSS/OPML origins before approval.

## `.data` contents

Generated data lives under:

```text
.data/rss/local/
  catalog.json       # bounded metadata catalog for date-only queries and change detection
  items/             # LanceDB vector store opened as ~/rss/local/items
```

`catalog.json` schema:

```ts
interface Catalog {
  version: 1;
  updatedAt?: string;
  items: Array<{
    id: string;
    title: string;
    feedUrl: string;
    feedTitle?: string;
    link?: string;
    guid?: string;
    publishedAt?: string;
    summary?: string;
    hash: string;
    indexedAt: string;
  }>;
}
```

LanceDB rows use the same stable `id` plus `vector`, `title`, `excerpt`, `feedUrl`, `feedTitle`, `link`, `guid`, `publishedAt`, and `indexedAt` props. Data is derived from configured external feeds and can be regenerated by the daily cron, subject to feed retention and availability.

## Safe `.data` inspection

- Prefer read-only inspection through Metidos diagnostics or by copying `.data` aside while the plugin is stopped.
- Do not print large catalogs or copied feed excerpts into chat or logs.
- The plugin intentionally stores summaries/excerpts, not full feed bodies, but external feed content can still be sensitive or private if users index authenticated/private feed URLs in an approved environment.
- LanceDB data is derived and should normally be treated as rebuildable.

## Safe `.data` repair

1. Stop or disable the plugin before mutating `.data`.
2. Back up `.data/rss/local/` or use Metidos Reset Plugin Data.
3. For small metadata issues, edit only `catalog.json` and preserve `version: 1` plus the item schema above.
4. For corrupt LanceDB data, prefer deleting the local `items/` directory and letting the cron rebuild changed/new rows. If `catalog.json` still marks items unchanged, delete `catalog.json` too so all currently visible feed items are re-embedded.
5. Restart/retry the plugin and watch diagnostics for missing settings, denied network origins, XML parse failures, or embedding-provider failures.

## Reset behavior

Metidos Reset Plugin Data moves `.data` to `.data-bak-{timestamp}`, recreates `.data`, copies any `seed/**` files, and restarts/reloads the plugin. This plugin has no seed data. After reset, the configured `list_url` Plugin Setting remains in Metidos settings, but RSS catalog and LanceDB vectors are gone until the next cron run rebuilds them from currently available feed entries.

## Secrets and logs

This plugin declares no secret settings or env vars. Feed URLs in `list_url` may still reveal private interests or private feed tokens if users put tokens in URLs. Prefer token-free public feed URLs or feeds whose access control is not encoded in the URL.

Logs are generic operational diagnostics: skipped users, changed-item counts, failed URL counts, hashed failed-URL identifiers, and bounded fetch/parse error messages. Do not add raw feed item bodies, query text, embedding input, or secret-bearing URLs to logs.

## Embeddings and vector search

This plugin consumes embeddings and stores LanceDB vectors. It does not provide embedding models.

- `metidos:can_embed` is required because the daily cron embeds changed/new feed-item text and `rss_query` embeds `q` for semantic search.
- `metidos:lancedb` and `storage:write` are required because the plugin opens `~/rss/local/items` and upserts vector rows.
- The embedded text is a bounded concatenation of feed item title, summary/content excerpt, and feed title.
- Embedding payloads include only purpose strings, stable item ids, and hashed source-url hints; they must not include raw feed bodies or secrets.
- If no runtime embedding model is configured for the user, indexing and semantic query calls fail with host embedding diagnostics.
- Date-only `rss_query` uses `catalog.json` because Plugin System v1 LanceDB exposes vector search, not full table scans.

## Context notes

- `refresh_feeds` is a local cron. It runs once on the configured schedule and reads the `list_url` Plugin Setting.
- `rss_query` is a thread tool and reads the shared local RSS store.
- Project `./` file APIs are not used and should not be added.
- Network fetches use `network.allow: ["https://**/**"]` and require the manifest `unsafe` permission; safe outbound/private-network protections still apply.
- OPML child feed URLs are also subject to the same all-domain HTTPS review risk. The plugin must parse RSS/OPML as untrusted XML and must not trust feed item content.
