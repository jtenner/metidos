# AGENTS for Telegram Notification Provider

## Purpose

First-party notification provider plugin for sending Metidos notifications through the Telegram Bot API `sendMessage` endpoint.

The plugin registers one notification provider, `telegram`, and may contact only `https://api.telegram.org/bot*/sendMessage` through `metidos.fetch`.

## Credential reuse decision

The notification provider may reuse the same deployment-level `TELEGRAM_BOT_TOKEN` environment variable as Telegram ingress. This is the only cross-plugin reuse path. Plugin Settings remain plugin-scoped secrets: this provider reads only its own `bot_token` setting and must not expose it for Telegram ingress to consume implicitly.

Use this precedence for notifications:

1. this plugin's `bot_token` Plugin Setting
2. `TELEGRAM_BOT_TOKEN`

If both Telegram plugins are enabled and should share a bot, prefer setting `TELEGRAM_BOT_TOKEN` once at deployment time. Otherwise copy the token into each plugin's own settings deliberately.

## Source layout

- `metidos-plugin.json`: v1 manifest, network allowlist, env/settings declarations, and notification provider declaration.
- `index.ts`: plugin entry point and Telegram send callback.
- `AGENTS.md`: operator and agent guidance.
- `.data/`: not used by this plugin; generated by Metidos if present.
- `.logs/`: generated only if plugin logging is enabled; do not commit.
- `.data-bak-*`: generated reset backups; do not commit.

Root `node_modules/` is forbidden. Keep imports local or `@metidos/plugin-api` only.

## Validation

From the Metidos repository root:

1. Validate `core_plugins/telegram_notification_provider/metidos-plugin.json` against `docs/metidos-plugin.schema.json`.
2. Run targeted plugin manifest/runtime tests when available.
3. Confirm no root `node_modules/` exists.
4. Confirm imports are local or `@metidos/plugin-api`.

## `.data` contents

This plugin intentionally does not use plugin-owned `.data`. If `.data` exists after experimentation, inspect it read-only before deleting it while the plugin is stopped.

## Safe `.data` inspection

- Prefer read-only inspection.
- Do not copy or print secret-bearing files unless needed for repair.
- Do not edit `.data` while the plugin sidecar is running.

## Safe `.data` repair

There are no documented repairable `.data` files. Prefer Metidos Reset Plugin Data or deleting unused `.data` while the plugin is stopped.

## Reset behavior

Reset Plugin Data moves `.data` to `.data-bak-{timestamp}`, recreates `.data`, and restarts/reloads the plugin. Because this plugin has no seed data and stores no runtime state, reset does not change Telegram configuration. Reconfigure settings in Metidos if needed.

## Secrets and logs

- `TELEGRAM_BOT_TOKEN`, this plugin's `bot_token` Plugin Setting, and `chat_id` can identify or control notification delivery.
- This plugin's `bot_token` setting is scoped to this plugin. Do not make Telegram ingress read or import it implicitly.
- Telegram chat IDs are intentionally setting-only; do not add `TELEGRAM_CHAT_ID` or any other chat-id environment variable.
- Do not log bot tokens, chat IDs, request bodies, full Telegram URLs, or provider payloads.
- v1 does not promise automatic redaction of plugin-authored logs, thrown errors, or receipts.

## Context notes

Notifications require the `enabled` Plugin Setting to be true and a single `chat_id` setting. Bot token precedence is the `bot_token` Plugin Setting, then `TELEGRAM_BOT_TOKEN`.

## Embeddings and vector search

This plugin does not currently provide embedding models, call `metidos.embeddings.embed(...)`, or store LanceDB vectors. If future changes add semantic search or memory, update the manifest and this file together: embedding providers use `metidos:provides_embeddings`; embedding consumers use `metidos:can_embed`; vector storage uses `metidos:lancedb` plus `storage:write` and must stay scoped to plugin-owned `~/` data.
