# AGENTS for Telegram Ingress

## Purpose

First-party request ingress plugin for polling Telegram Bot API `getUpdates` and forwarding text messages into Metidos ingress. The plugin also supports `reply_to_source`; replies are sent only through Telegram Bot API `sendMessage` to the verified source chat derived from Metidos ingress context.

The plugin registers one ingress source, `telegram`, and may contact only:

- `https://api.telegram.org/bot*/getUpdates`
- `https://api.telegram.org/bot*/getFile`
- `https://api.telegram.org/bot*/sendMessage`
- `https://api.telegram.org/file/bot*/**`

## Credential reuse decision

Telegram ingress may reuse the same deployment-level `TELEGRAM_BOT_TOKEN` environment variable as the Telegram notification provider. This is the only cross-plugin reuse path. Plugin Settings remain plugin-scoped secrets: ingress reads only its own `bot_token` setting and must not read or import `telegram_notification_provider` settings implicitly.

Use this precedence for ingress requests and replies:

1. `TELEGRAM_BOT_TOKEN`
2. this plugin's `bot_token` Plugin Setting

If both Telegram plugins are enabled and should share a bot, prefer setting `TELEGRAM_BOT_TOKEN` once at deployment time. Otherwise copy the token into each plugin's own settings deliberately.

## Source layout

- `metidos-plugin.json`: v1 manifest, network allowlist, env/settings declarations, and ingress source declaration.
- `index.ts`: plugin entry point, Telegram long-poll mapper, and source reply callback.
- `AGENTS.md`: operator and agent guidance.
- `.data/`: not used by this plugin; generated by Metidos if present.
- `.logs/`: generated only if plugin logging is enabled; do not commit.
- `.data-bak-*`: generated reset backups; do not commit.

Root `node_modules/` is forbidden. Keep imports local or `@metidos/plugin-api` only.

## Validation

From the Metidos repository root:

1. Validate `core_plugins/telegram_ingress/metidos-plugin.json` against `docs/metidos-plugin.schema.json`.
2. Run targeted plugin manifest/startup tests when available.
3. Confirm no root `node_modules/` exists.
4. Confirm imports are local or `@metidos/plugin-api`.

Manual smoke test after approval:

1. Configure `TELEGRAM_BOT_TOKEN` or this plugin's `bot_token` Plugin Setting. The env var intentionally matches the Telegram notification provider so one deployment secret can serve both plugins without sharing Plugin Settings.
2. Approve and enable the plugin.
3. Enable or bind the `telegram` ingress source according to Metidos ingress settings.
4. Send a plain text message to the bot from the Telegram chat/user being linked.
5. Confirm Metidos creates/updates the ingress thread and that `reply_to_source` sends a Telegram reply only when explicitly called.

## `.data` contents

This plugin intentionally does not use plugin-owned `.data`. Telegram polling cursor state is owned by Metidos ingress, not by plugin files. If `.data` exists after experimentation, inspect it read-only before deleting it while the plugin is stopped.

## Safe `.data` inspection

- Prefer read-only inspection.
- Do not copy or print secret-bearing files unless needed for repair.
- Do not edit `.data` while the plugin sidecar is running.

## Safe `.data` repair

There are no documented repairable `.data` files. Prefer Metidos Reset Plugin Data or deleting unused `.data` while the plugin is stopped.

## Reset behavior

Reset Plugin Data moves `.data` to `.data-bak-{timestamp}`, recreates `.data`, and restarts/reloads the plugin. Because this plugin has no seed data and stores no runtime state, reset does not change Telegram configuration or Metidos-owned ingress cursors/bindings. Reconfigure plugin settings in Metidos if needed.

## Secrets and logs

- `TELEGRAM_BOT_TOKEN` and this plugin's `bot_token` Plugin Setting can control the Telegram bot.
- The `bot_token` setting is scoped to this plugin. Do not read or expose the Telegram notification provider's `bot_token` setting.
- Telegram chat IDs, user IDs, update IDs, and message IDs are external provider identifiers. Treat them as sensitive operational metadata.
- Do not log bot tokens, message bodies, full Telegram URLs, provider payloads, or reply payloads.
- This plugin requests `log:write` for bounded ingress diagnostics. Logs may include Telegram chat IDs, user IDs, update IDs, and message IDs while debugging routing/polling. Do not add raw message text or token-bearing values to logs.
- v1 does not promise automatic redaction of plugin-authored thrown errors or ingress prompt text.

## Context notes

The plugin receives no Metidos user context and cannot choose arbitrary reply destinations. `reply_to_source` resolves the plugin/source and external Telegram chat from Metidos-owned verified ingress context. The source polls Telegram messages, maps text/captions and supported image attachments, and skips bot-authored messages, unsupported non-text messages, and malformed updates while advancing the Telegram update cursor to avoid replay loops.

## Embeddings and vector search

This plugin does not currently provide embedding models, call `metidos.embeddings.embed(...)`, or store LanceDB vectors. If future changes add semantic search or memory, update the manifest and this file together: embedding providers use `metidos:provides_embeddings`; embedding consumers use `metidos:can_embed`; vector storage uses `metidos:lancedb` plus `storage:write` and must stay scoped to plugin-owned `~/` data.
