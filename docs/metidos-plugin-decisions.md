# Metidos Plugin System v1 decision ledger

This ledger records locked v1 decisions so implementation workers do not rediscover or renegotiate Plugin System v1 behavior.

| ID | Decision | Rationale | Implementation consequence |
| --- | --- | --- | --- |
| PLUG-001 | Local-operator approval is the v1 trust boundary. | Plugins are local, trusted extensions; Metidos can review and limit capabilities but cannot prove approved code is semantically safe. | Never execute plugin code before local-operator approval. UI copy must avoid promising data-loss prevention for data a plugin is allowed to read. |
| PLUG-002 | Plugins are not UI or backend RPC extensions in v1. | The first version must preserve Metidos app architecture and avoid unbounded extension surfaces. | Do not add React plugin panels, frontend component injection, plugin-defined RPC routes, or arbitrary server hooks. |
| PLUG-003 | Use exactly one sidecar process per activated plugin. | Per-plugin isolation simplifies crash ownership, diagnostics, approval state, and lifecycle control. | Lifecycle manager starts one sidecar for each approved active plugin and marks only that plugin failed/degraded on startup or runtime failure. |
| PLUG-004 | Execute plugin code in QuickJS, not in the main Bun runtime. | QuickJS provides a smaller injected surface for plugin callbacks and avoids exposing host globals by default. | Sidecars build with Bun, load the artifact into QuickJS, and inject only the Metidos plugin API. |
| PLUG-005 | Use custom typed JSON RPC over stdio between host and sidecar. | A host-owned protocol keeps plugin capabilities explicit without exposing plugin-defined backend RPC. | stdout is protocol only; stderr is diagnostics. Messages use typed envelopes with size limits and correlation IDs. |
| PLUG-006 | Do not install runtime dependencies for plugins. | Runtime package installation creates supply-chain, lifecycle, and reproducibility problems. | `node_modules/` in a plugin root is activation-blocking. Bare imports are rejected except `@metidos/plugin-api`. |
| PLUG-007 | `AGENTS.md` is required for every plugin. | Operators and future agents need plugin-owned guidance for safe inspection, repair, validation, and `.data` handling. | Missing or unreadable `AGENTS.md` blocks activation and affects the review hash. |
| PLUG-008 | Disabled plugins require a Metidos restart for full removal from a running runtime. | Hot-unregistering tools, providers, crons, and callbacks is complex and error-prone for v1. | Disable records disabled/restart-required state and UI must tell the local operator to restart. Do not promise immediate removal of already-registered capabilities. |
| PLUG-009 | Thread access controls tool visibility only. | Access groups are a usability and runtime exposure control, not a permission grant. | Host APIs still check globally approved plugin permissions and context rules. Enabling a thread group cannot add permissions. |
| PLUG-010 | Deterministic review hashing excludes only plugin-owned mutable data/log paths. | The local operator must re-review code/config changes while normal data/log churn should not invalidate approval. | Hash sorted `relative path + NUL + content hash`, include source/support/seed files, exclude `.data/**`, `.data-bak-*/**`, and `.logs/**`. |
| PLUG-011 | Plugin `~/` data is separate from project `./` access. | Persistent plugin state and project-file operations have different safety and context rules. | `~/` maps to plugin `.data` and uses storage permissions. `./` is available only in thread tool contexts and additionally uses manifest file allowlists. |
| PLUG-012 | First-activation seeding is one-shot except explicit reset. | Silent reseeding after deletion could recreate stale data unexpectedly. | Copy `seed/**` only on first activation or Reset Plugin Data; preserve an activated-once flag across restarts. |
| PLUG-013 | Logging is local-operator opt-in and separate from sidecar stderr. | Plugin logs can contain sensitive plugin-authored text; stderr is runtime diagnostics. | `metidos.log` requires `log:write` and logging enabled. Stderr always streams locally and keeps last 200 diagnostic lines. |
| PLUG-014 | Provider and notification provider registration are initialization-only. | Provider surfaces should be stable after startup and reviewable from manifest declarations. | Disable provider registration APIs in tool, cron, GC, provider execution, and notification provider execution contexts. |
| PLUG-015 | Terminal create/kill require `unsafe`; terminal APIs are never available in cron. | Terminal access is a high-impact local capability and cron has no interactive operator context. | `terminal:create` and `terminal:kill` require `unsafe`; all terminal APIs throw context errors in cron. |
| PLUG-016 | Calendar/event delete always requires confirmation. | Deletion should remain user-confirmed even for approved plugins. | Delete APIs fail in cron and other contexts where confirmation is unavailable. |
| PLUG-017 | Plugin SQLite is only for plugin `.data`. | SQLite should support plugin-local state without exposing host or project databases. | `sqlite` requires `storage:write` and can open only paths under `~/`; never `./` or the Metidos app DB. |
| PLUG-018 | Network allowlists are URL boundaries, not semantic DLP. | URL patterns are understandable and reviewable, but do not prove safe intent. | `network:fetch` requires `network.allow`; every redirect hop must match. Docs and UI should avoid overclaiming. |
| PLUG-019 | Secret values are not automatically redacted from plugin-authored output in v1. | Full output redaction is complex and unreliable. | Avoid intentional secret logging in docs/examples; do not promise automatic redaction for plugin logs, tool results, or provider output. |
| PLUG-020 | Model provider identities are stable composite keys. | Stable identities preserve active selections when a model remains present across refreshes and restarts. | Use `plugin_id/provider_id/configuration_id/model_id`; disappearing models are removed from the catalog instead of being synthesized as cached fallback rows. |
| PLUG-021 | Python plugin entrypoints are part of Plugin System v1 through the safe Pyodide host. | The schema, authoring guide, and copyable `python_hello_tool` example already expose `.py` entrypoints, and keeping them in v1 avoids a hidden experimental fork of the plugin contract. Pyodide preserves the no-host-Python boundary while sharing the reviewed manifest, sidecar lifecycle, permissions, and host API checks. | Keep QuickJS as the TypeScript/JavaScript adapter and deepen the multi-adapter runtime path for `.py` manifests. Python docs and tests must describe the safe Pyodide host, not host Python, and runtime changes must avoid adding raw Python filesystem, network, process, or package-install access. |
| PLUG-022 | Embedding providers and embedding consumers use separate permissions. | Providing an embedding model and embedding plugin-owned/user content are different trust decisions. Provider plugins should not automatically gain the ability to call the host embedding API or store vectors. | Provider plugins that expose embedding models declare `metidos:provides_embeddings` and implement `embed(context, request)`. Consumer plugins that call `metidos.embeddings.embed(...)` declare `metidos:can_embed`. Vector storage additionally requires `metidos:lancedb` and `storage:write`. |
| PLUG-023 | Plugin LanceDB storage is plugin-owned derived data by default. | Semantic indexes and memories need persistent vectors without exposing project paths or app databases. Treating vector stores like plugin `.data` keeps review, reset, quota, and repair behavior consistent. | `metidos.lancedb.open(path)` accepts only `~/` plugin data paths, never `./` project paths. Plugin `AGENTS.md` files must document vector store paths, row schemas, regeneration/reset behavior, and sensitive-content rules. |
| PLUG-024 | Plugin step-up is required for actions that approve or run plugin code, not for risk-reducing maintenance. | Disabling a plugin reduces sandboxed runtime exposure, and resetting plugin-owned `.data` is a bounded recovery operation protected by explicit destructive confirmation. Enabling, re-approving, retrying, and running GC can approve code, reactivate approved code, or invoke plugin callbacks. | Require recent step-up for Enable, Re-approve Plugin, Retry Plugin, and Run Plugin GC. Require a local-operator session, but not recent step-up, for Disable, Review Plugin Changes, Open `.data`, Open `.logs`, and Reset Plugin Data. |
