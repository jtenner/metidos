# Agents Todo

This file is the execution backlog for replacing `@openai/codex-sdk` with Pi in `jt-ide`.

## Rules

- Remove completed todo items from this document altogether. Do not leave completed items in place as checked, archived, or struck through entries.
- Keep the `Risks` and `Blockers` sections current before adding, reordering, or splitting slices.
- Add new slices only when they clearly map back to the migration requirements in [the research document](./docs/2026-04-09-pi-coding-agent-migration-research.md).

## Risks

- Hosted-provider compatibility risk. The Pi runtime adapter now runs real Jolt thread lifecycles through Bun successfully, but the committed coverage is still probe-backed plus focused smoke tests. Real hosted-provider auth, transport quirks, and long-running tool sessions still need follow-up coverage. See [1. Runtime Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#1-runtime-integration), [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), and [Recommended Phased Plan](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).
- Safety semantics risk. `unsafeMode=false` now has a temporary Pi meaning in code: worktree-scoped `read`, `ls`, `find`, `grep`, `edit`, and `write`, with `bash` disabled. That is more explicit than before, but it is still not equivalent to Codex sandbox/network semantics and should not be treated as final product policy. See [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), [No built-in Codex-style sandbox policy](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-codex-style-sandbox-policy), and [6. Access Controls and Unsafe Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#6-access-controls-and-unsafe-mode).
- Feature parity risk. GitHub tools, sub-agents, plan mode, web search, and Jolt MCP-backed tools are not native Pi features, so parity requires explicit custom implementation and careful scope control. See [Pi Non-Goals That Matter for Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#pi-non-goals-that-matter-for-jolt), [7. GitHub Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#7-github-integration), and [8. Agents and Plan Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#8-agents-and-plan-mode).
- Transcript regression risk. The live Pi runtime now restores `chat`, `reasoning`, `command`, and generic `tool_call` rows, but `file_change`, `web_search`, and exact Codex-shaped tool parity are still missing. See [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), [Current Codex-Specific Surface In Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#current-codex-specific-surface-in-jolt), and [5. Transcript/Event Mapping](./docs/2026-04-09-pi-coding-agent-migration-research.md#5-transcriptevent-mapping).
- Persistence split-brain risk. The runtime now uses deterministic Pi session directories under Jolt app data and temporarily stores the Pi session id in `codexThreadId`, which is pragmatic but still leaves a compatibility-shaped persistence layer in place until DB03 lands. See [RT02 implementation status in `jt-ide`](./docs/2026-04-09-pi-coding-agent-migration-research.md#rt02-implementation-status-in-jt-ide), [3. Pi already has a full session system](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-pi-already-has-a-full-session-system), and [2. Persistence and Thread Identity](./docs/2026-04-09-pi-coding-agent-migration-research.md#2-persistence-and-thread-identity).
- UI and model semantics risk. Current model selection, reasoning controls, and access toggles are Codex-shaped, so the frontend will need semantic changes rather than a transport swap. See [Current Codex-Specific Surface In Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#current-codex-specific-surface-in-jolt), [3. Provider/Model UI Migration](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-providermodel-ui-migration), and [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix).

## Blockers

- Choose the GitHub integration strategy: `gh` wrapper, REST/GraphQL extension, or hybrid. This blocks parity for the existing `GitHub` thread access toggle. See [No built-in GitHub connector](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-github-connector) and [7. GitHub Integration](./docs/2026-04-09-pi-coding-agent-migration-research.md#7-github-integration).
- Decide acceptable parity for agents, plan mode, and transcript diff cards before implementation begins. Otherwise the migration will drift between “exact Codex parity” and “Pi-native simplification” mid-stream. See [5. Transcript/Event Mapping](./docs/2026-04-09-pi-coding-agent-migration-research.md#5-transcriptevent-mapping), [8. Agents and Plan Mode](./docs/2026-04-09-pi-coding-agent-migration-research.md#8-agents-and-plan-mode), and [Open Questions](./docs/2026-04-09-pi-coding-agent-migration-research.md#open-questions-that-must-be-answered-before-full-migration).

## Todo Items

- [DB][03] - Persist Pi Session Identity on Jolt Threads
Add the minimum schema and runtime changes needed for Jolt threads to reference Pi sessions, session files, and any future tree entry identity without replacing SQLite as the app-level store. This should implement the storage direction in [3. Pi already has a full session system](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-pi-already-has-a-full-session-system), [2. Persistence and Thread Identity](./docs/2026-04-09-pi-coding-agent-migration-research.md#2-persistence-and-thread-identity), and [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix).

- [MD][04] - Replace the Static Codex Model Catalog
Move model discovery from hardcoded Codex/xAI lists to Pi’s provider and model registry so the backend can expose multiple providers and endpoints through one normalized catalog. This slice should implement the changes called for in [2. Pi already solves the “multiple endpoints, same interface” problem better than current Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#2-pi-already-solves-the-multiple-endpoints-same-interface-problem-better-than-current-jolt), [3. Provider/Model UI Migration](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-providermodel-ui-migration), and [Phase 2](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [UI][05] - Rework Model and Thinking Controls for Pi
Update the frontend and persistence model so thread settings use provider plus model identity and Pi-compatible thinking controls instead of Codex-only model ids and `reasoningEffort` semantics. This slice should follow [3. Provider/Model UI Migration](./docs/2026-04-09-pi-coding-agent-migration-research.md#3-providermodel-ui-migration), [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix), and [What Jolt Still Must Build By Hand Even If It Uses Pi](./docs/2026-04-09-pi-coding-agent-migration-research.md#what-jolt-still-must-build-by-hand-even-if-it-uses-pi).

- [EV][06] - Build the Pi-to-Jolt Event Projection Layer
Map Pi session events into Jolt’s persisted message records, run status updates, and websocket payloads without pretending Pi emits the same item types as Codex. This slice is defined by [Current Codex-Specific Surface In Jolt](./docs/2026-04-09-pi-coding-agent-migration-research.md#current-codex-specific-surface-in-jolt), [5. Transcript/Event Mapping](./docs/2026-04-09-pi-coding-agent-migration-research.md#5-transcriptevent-mapping), and [Phase 1](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [TL][07] - Restore Transcript Parity for Commands, Tools, and File Changes
Decide and implement the minimum acceptable parity for command rows, tool-call rows, reasoning rows, and file-change cards, including any synthesized diffs required for Pi `write` operations. This slice should directly address [5. Transcript/Event Mapping](./docs/2026-04-09-pi-coding-agent-migration-research.md#5-transcriptevent-mapping), [Migration Requirement Matrix](./docs/2026-04-09-pi-coding-agent-migration-research.md#migration-requirement-matrix), and [Open Questions](./docs/2026-04-09-pi-coding-agent-migration-research.md#open-questions-that-must-be-answered-before-full-migration).

- [JT][08] - Port Jolt MCP Tools into Pi Extensions and Custom Tools
Replace the Jolt MCP sidecar path with Pi-native tools or extensions for thread management, cron management, context setting, and the untrusted JS runner so the agent no longer depends on a self-hosted MCP bridge. This slice should implement [No MCP](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-mcp), [4. Porting Jolt Tools Out of MCP and Into Pi](./docs/2026-04-09-pi-coding-agent-migration-research.md#4-porting-jolt-tools-out-of-mcp-and-into-pi), and [Phase 4](./docs/2026-04-09-pi-coding-agent-migration-research.md#recommended-phased-plan).

- [SA][09] - Define and Implement Safe versus Unsafe Tool Policy

This slice should close the largest semantics gap described in [No built-in Codex-style sandbox policy](./docs/2026-04-09-pi-coding-agent-migration-research.md#no-built-in-codex-style-sandbox-policy), [6. - [GH][10] - Implement a Pi-Native GitHub Tool Pack
