Plugin System v1

Plugins are local, review-first extensions.

Metidos plugins are folders installed under App Data, discovered without execution, reviewed by the Local Operator, and only run after approval.

Read plugin docs Build a plugin

Plugin System v1 is experimental and local-operator-approved. APIs and review flows may change before a stable public release.

Safety model

Review before runtime

Discover first

Metidos lists plugin folders and reads manifest/runtime summaries before executing plugin code.

Review capabilities

The Local Operator reviews permissions, access groups, settings, env declarations, allowlists, and source changes.

Approve current code

Approval is tied to a deterministic review hash; source or manifest changes require review again.

Run in bounded contexts

Approved plugins run in sidecars while host APIs enforce declared permissions and allowlists.

Approval flow

From folder to sidecar

The page intentionally stays high level. Use the repository docs for exact schema, commands, and review UI details.

  1. 1. Copy a plugin folder under APP_DATA/plugins/{plugin_id}/.
  2. 2. Metidos discovers the folder without executing code.
  3. 3. Settings → Plugins surfaces manifest details, review hash, source changes, and declared capabilities.
  4. 4. The Local Operator reviews and approves the current hash after step-up authentication when required.
  5. 5. Approved code starts in a per-plugin sidecar and registers only declared capabilities.
  6. 6. Threads can opt into plugin Access Groups when appropriate.
  7. 7. Changes outside excluded runtime paths invalidate approval and require re-review.
  8. 8. Disable, reset data, or remove the plugin when no longer trusted or needed.

Capabilities

Permissions and Access Groups are different controls

A manifest Permission grants a host capability to approved plugin code. An Access Group controls which plugin tools are visible to selected Threads; it does not grant host APIs by itself.

Permission reference
  • Plugin-scoped storage and logging
  • Project file access through explicit allowlists
  • Network fetch and websocket allowlists
  • SQLite and vector/embedding storage
  • Notification and model provider registration
  • Agent tools and prompt injection
  • Plugin crons
  • Request ingress and reset callbacks

Local data

Separate reviewed source from runtime state

Source and review files include metidos-plugin.json, AGENTS.md, manifest main, examples, and seed files.

Runtime state belongs in .data/, .data-bak-*, and .logs/ under the plugin folder.

Settings and secrets live in Metidos App Data. Secret settings are encrypted and should not appear in diagnostics, screenshots, examples, or copied runtime state.

Operator checklist

Approve narrowly, revisit often

  • Install plugins only from sources you trust enough to review.
  • Read the manifest, AGENTS.md, and changed source before approval.
  • Verify every permission and allowlist is necessary.
  • Keep access groups narrow and enable them only for threads that need them.
  • Re-review after source or manifest changes.
  • Disable or remove plugins that are no longer needed.
