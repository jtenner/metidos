## Risks

- Billing-precedence risk. Jolt currently hides `openai-codex` in the catalog and resolves overlapping raw model ids such as `gpt-5.4` toward plain `openai`, which can route users into API billing instead of ChatGPT-plan Codex. See [Bottom Line](./docs/2026-04-09-codex-via-pi-wiring.md#bottom-line), [Current Jolt State](./docs/2026-04-09-codex-via-pi-wiring.md#current-jolt-state), and [5. Fix runtime provider selection and billing precedence](./docs/2026-04-09-codex-via-pi-wiring.md#5-fix-runtime-provider-selection-and-billing-precedence).
- Selector-ambiguity risk. A flat picker still makes `openai-codex:gpt-5.4` and `openai:gpt-5.4` look like the same choice unless provider is the first selection step and reasoning effort is driven from the final model choice. See [1. Define the selector as `Provider -> Model -> Reasoning effort`](./docs/2026-04-09-codex-via-pi-wiring.md#1-define-the-selector-as-provider---model---reasoning-effort), [Risks](./docs/2026-04-09-codex-via-pi-wiring.md#risks), and [CD02 - Build the three-level model selector](./docs/2026-04-09-codex-via-pi-wiring.md#cd02---build-the-three-level-model-selector).
- Storage-parity risk. OpenAI documents Codex credential storage in `~/.codex/auth.json` or an OS keyring, while Pi documents `auth.json` storage under the configured agent directory. If Jolt needs keyring parity or direct reuse of the Codex auth cache, that is extra implementation beyond the recommended first pass. See [6. Decide on auth-storage parity](./docs/2026-04-09-codex-via-pi-wiring.md#6-decide-on-auth-storage-parity), [Risks](./docs/2026-04-09-codex-via-pi-wiring.md#risks), and [Recommendation](./docs/2026-04-09-codex-via-pi-wiring.md#recommendation).
- Browser-login risk. Pi's OAuth flow is backend-only, and Jolt currently has no provider-auth RPC or browser UI. The app must build that orchestration itself. See [3. Add backend provider-auth procedures](./docs/2026-04-09-codex-via-pi-wiring.md#3-add-backend-provider-auth-procedures), [4. Add browser provider-auth UI](./docs/2026-04-09-codex-via-pi-wiring.md#4-add-browser-provider-auth-ui), and [CD03 - Add backend `openai-codex` auth procedures](./docs/2026-04-09-codex-via-pi-wiring.md#cd03---add-backend-openai-codex-auth-procedures).
- Headless-flow risk. Codex publicly documents device-code authentication and localhost-callback recovery, but Pi's built-in OpenAI Codex docs do not document the same end-user recovery story. Jolt should not assume it gets that parity for free. See [3. Public Codex authentication behavior is documented](./docs/2026-04-09-codex-via-pi-wiring.md#3-public-codex-authentication-behavior-is-documented), [7. Add diagnostics and recovery](./docs/2026-04-09-codex-via-pi-wiring.md#7-add-diagnostics-and-recovery), and [Risks](./docs/2026-04-09-codex-via-pi-wiring.md#risks).
- Policy-scope risk. OpenAI documents that ChatGPT-authenticated Codex usage follows ChatGPT workspace controls and retention, while API-key usage follows API org policy instead. Jolt must make the chosen auth mode and provider obvious to users. See [3. Public Codex authentication behavior is documented](./docs/2026-04-09-codex-via-pi-wiring.md#3-public-codex-authentication-behavior-is-documented), [4. Add browser provider-auth UI](./docs/2026-04-09-codex-via-pi-wiring.md#4-add-browser-provider-auth-ui), and [CD04 - Add browser provider-auth UI](./docs/2026-04-09-codex-via-pi-wiring.md#cd04---add-browser-provider-auth-ui).

## Blockers

Exact Codex credential-store parity only becomes a blocker if Jolt decides it must ship keyring support or direct reuse of `~/.codex/auth.json` before the first working implementation. See [Blockers](./docs/2026-04-09-codex-via-pi-wiring.md#blockers), [6.
